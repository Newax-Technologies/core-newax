# NEWAX Engineering Learning Automation Standard

## Status

Mandatory extension of the NEWAX Engineering Learning System.

## Purpose

This standard converts engineering learning from a voluntary retrospective into an observable, evidence-backed, and merge-blocking control system.

The system must:

- Capture failed GitHub Actions jobs and logs.
- Capture local command failures executed through the governed command wrapper.
- Accept planning, communication, external-tool, and manual mistakes through workflow dispatch, repository dispatch, or the structured issue form.
- Classify failures into controlled categories.
- Generate root-cause candidates with stated confidence.
- Detect exact recurrences and likely shared root causes.
- Preserve machine evidence independently from explanatory prose.
- Require confirmation for root causes that cannot be proven deterministically.
- Block a pull request that declares `Learning outcome: none` despite observable failure evidence.
- Require a new or existing ledger entry when learning evidence exists.
- Verify machine-supported claims against the original workflow run, failed job, logs, commit, and fingerprint.
- Require resolution evidence before a non-deterministic root cause may be treated as confirmed.

## 1. Evidence Boundary

Automation can inspect only observable evidence.

Observable sources include:

- GitHub Actions workflow runs and job logs.
- Pull-request commits, metadata, reviews, and linked issues.
- Local commands executed through `tooling/run-engineering-command.mjs`.
- Events submitted through `workflow_dispatch`.
- Events submitted through `repository_dispatch` using `engineering_mistake`.
- Structured engineering-mistake issues.

A private thought, unlogged terminal command, private chat, offline decision, or external-tool action that emits no event cannot be discovered by repository software. Such sources must use the dispatch or issue intake. Claiming otherwise would invent observability that does not exist.

## 2. Automatic CI Failure Intake

The `Engineering Failure Intake` workflow runs after monitored workflows complete.

For failed, timed-out, or action-required runs it must:

1. Read the workflow-run payload.
2. Fetch failed jobs and failed steps.
3. Fetch the complete job logs.
4. Normalize volatile timestamps and identifiers.
5. Classify the failure using the machine-readable catalog.
6. Generate a stable fingerprint.
7. Find exact recurrences and root-cause matches.
8. Create or update an `engineering-learning` issue.
9. Preserve the run, job, step, commit, pull request, evidence links, category, candidate cause, confidence, and prevention guidance.

Cancelled runs are not automatically treated as mistakes because branch synchronization and concurrency controls can cancel healthy work.

## 3. External and Non-PR Intake

The `Engineering External Intake` workflow accepts:

- Local-command failures.
- External-tool errors.
- Planning and decision-quality mistakes.
- Communication and polling waste.
- Manual observations.

External systems and AI agents should send a `repository_dispatch` event of type `engineering_mistake` immediately after an observable mistake.

Humans may use the workflow-dispatch form or the engineering-mistake issue form.

Local command failures generated by `pnpm verify` are queued in:

```text
.newax/engineering-events.ndjson
```

The queue is excluded from Git and can be submitted with:

```bash
pnpm learning:flush
```

## 4. Classification and Root-Cause Confidence

The catalog is:

- `docs/verification/engineering-learning-catalog.json`

Each known root cause defines:

- Stable root-cause ID.
- Ledger entry.
- Category.
- Error signatures.
- Confidence.
- Whether the signature supports deterministic machine classification.
- Root-cause statement.
- Unsuccessful method.
- Successful method.
- Prevention control.

Classification states are:

### Candidate

The failed stage or log pattern suggests a cause, but the machine cannot prove it.

### Machine-supported

A deterministic catalog signature matches the original workflow evidence and the reconciliation check reproduces the same fingerprint and root-cause ID.

### Confirmed

The cause is supported by evidence and a completed resolution record. Non-deterministic causes require confirmation rather than being promoted by confidence theatre.

The system must never label an unknown or heuristic cause as certain.

## 5. Duplicate and Recurrence Detection

The system compares:

- Exact fingerprint.
- Source event.
- Pull request.
- Stable root-cause ID.

An exact same occurrence updates the existing issue with recurrence evidence.

A new occurrence with the same root-cause ID receives its own issue and links the matching root-cause occurrence. This preserves occurrence-specific evidence while still identifying shared causes.

Unknown causes remain separate until evidence supports consolidation.

## 6. Factual Verification

Machine-supported claims are verified by re-reading:

- Workflow run conclusion.
- Commit SHA.
- Failed job.
- Failed step.
- Job logs.
- Recomputed fingerprint.
- Recomputed root-cause classification.

The pull-request reconciliation check rejects mismatched machine evidence.

For non-deterministic causes, software cannot prove semantic causality from prose alone. The learning issue must therefore contain:

- Complete evidence.
- Confirmed root cause.
- Verified resolution status.
- Fix commit.
- Successful focused and complete verification.
- Reviewer confirmation.

The system verifies presence and consistency of this record. Independent technical review remains responsible for semantic truth where deterministic proof is impossible.

## 7. False `none` Prevention

Before review, the reconciliation check inspects:

- Failed workflow runs across all commits in the pull request.
- Linked engineering-learning issues.
- Machine and externally submitted events associated with the pull request.

`Learning outcome: none` is rejected whenever observable failure evidence exists.

When evidence exists, the pull request must use:

- `new` when a new root cause, method, or prevention control is introduced.
- `existing` when the occurrence maps to an existing ledger entry.

Both require an occurrence-specific learning issue.

## 8. Resolution Standard

A learning issue may be resolved only after recording:

- Confirmed cause.
- Affected scope.
- Failed method.
- Successful method.
- Prevention control.
- Fix commit.
- Focused verification.
- Complete applicable verification.
- Ledger link.

Changing the error message, skipping a gate, weakening strictness, or closing the issue without evidence is not resolution.

## 9. Operation-Intent Guard

Before a state-changing repository or GitHub operation, the actor should create an operation-intent record containing:

- Target type.
- Target identifier.
- Action name.
- Expected postcondition.
- Prohibited side effects.

Validate it with:

```bash
pnpm operation:validate path/to/operation-intent.json
```

The guard rejects mismatches such as `pull-request-metadata` combined with `update_issue`. External agents that cannot execute the repository command must implement the same target-action mapping before invoking a mutation.

A successful postcondition becomes the new source of truth and prohibits repeating the same mutation unless fresh evidence shows that the state changed again.

## 10. Merge Control

The following checks must be eligible to become required branch checks after this process PR is merged:

- Continuous Integration.
- Pull Request Governance.
- Database Registry Map where applicable.

Pull Request Governance validates both the PR structure and historical learning reconciliation.

A pull request with unreconciled observable failures must remain blocked.

## 11. Completion Claim

This automation reduces unrecorded and repeated mistakes. It does not create omniscience.

The correct claim is:

> Every observable failure source connected to the NEWAX learning intake is captured, classified, reconciled, and prevented from being hidden behind a false `none` declaration.

The incorrect claim is:

> The repository automatically knows every mistake made anywhere and always determines the true root cause without evidence or review.
