# ADR 0018: Build the Trusted Request Context Foundation

## 1. Decision Title

Build the first reusable Trusted Request Context foundation for NEWAX Core.

## 2. Status

Accepted

## 3. Date

2026-07-12

## 4. Context

NEWAX Core now has executable Organizations, People, Memberships, Access Control, Users, and Authentication foundations.

Authentication proves control of a user session but does not grant access to any organization. A person may belong to multiple organizations through separate memberships, and each membership may have a different permission set.

Application services currently accept request-context objects containing an actor user, organization, and permission codes. Those values must be generated by trusted infrastructure. They must never be copied from request bodies, query parameters, unverified headers, or client-provided claims.

The next controlled slice must establish one tenant-safe execution context that connects:

```text
Authentication
  -> Membership ownership and status
  -> Organization status
  -> Effective permission evaluation
  -> Organization-scoped application operation
```

## 5. Decision

NEWAX Core will provide a reusable `@newax/request-context` foundation package with:

- Authenticated account-context resolution.
- Organization-context resolution through an explicitly selected membership.
- Membership ownership and status validation.
- Organization status validation.
- Effective permission evaluation.
- Tenant-boundary integrity verification.
- Immutable permission sets.
- Permission authorization helpers.
- Conversion into existing module request-context contracts.
- A request-local context-store contract.
- A Node `AsyncLocalStorage` adapter.
- Authentication, Access Control, Prisma, and NestJS composition.
- Unit tests, documentation, changelog, version metadata, and Module Registry registration.

The module owns no database tables.

### 5.1 Account context

A valid session produces an account context containing:

```text
request_id
user_id
person_id
session_id
session_expires_at
```

An account context does not contain an organization, membership, role, or permission set.

A valid account session alone never grants organization access.

### 5.2 Organization selection

A caller may select only a membership identifier.

The resolver derives the organization identifier from the trusted membership record. It does not accept `organizationId` as request authority.

The selected membership must:

- Exist.
- Belong to the person attached to the authenticated session.
- Be active.
- Reference an active organization.

Missing memberships, memberships belonging to another person, suspended memberships, ended memberships, and inactive organizations return one concealed membership-unavailable result.

### 5.3 Permission evaluation

After membership validation, the resolver evaluates effective permissions for the membership.

The permission result must contain the same membership and organization identifiers resolved from the trusted membership directory. Any mismatch is an integrity failure.

The resulting permission codes are exposed through an immutable read-only set.

Roles remain permission bundles. Business logic continues to authorize through stable permission codes rather than role names.

### 5.4 Existing module compatibility

The trusted organization context can be converted into the request-context shape already consumed by foundation services:

```text
actorUserId = authenticated user ID
organizationId = organization derived from membership
permissionCodes = evaluated effective permission codes
```

This conversion must occur from trusted context. Controllers and clients must not construct these values directly.

### 5.5 Request-local storage

The API adapter uses Node `AsyncLocalStorage` to hold context during one explicitly established operation.

Rules:

- Missing context fails closed.
- Concurrent operations remain isolated.
- Context is not stored in process-global mutable state.
- Context is not reused across requests.
- Permission evaluation represents a request-scoped snapshot.

### 5.6 Platform context

Authenticated user sessions do not create platform context.

Omitting membership selection does not produce `organizationId = null`. Account context and platform context are different concepts.

Platform-wide execution remains restricted to separately trusted internal paths that will be designed in a later decision.

### 5.7 HTTP boundary

This decision does not expose public controllers, middleware, guards, or decorators.

Public HTTP integration remains deferred until the following are defined and tested together:

- Secure session-token transport.
- Trusted membership selection transport.
- Request throttling.
- CSRF policy.
- Trusted proxy handling.
- Security headers.
- Error mapping and response redaction.
- Durable audit persistence.
- Endpoint-specific permission guards.

## 6. Consequences

### Positive

- Session identity, tenant identity, and permissions are resolved through separate authoritative systems.
- A caller cannot select an organization without an active membership belonging to the authenticated person.
- Cross-tenant membership probing is concealed.
- Permission calculations are bound to the validated membership and organization.
- Existing modules can consume trusted contexts without redesigning their service contracts.
- Concurrent requests do not share context.
- Platform authority cannot be obtained by omitting organization data.

### Negative

- Every organization-scoped request requires membership and permission resolution unless a safe cache is introduced later.
- Permission changes during a request take effect on the next context resolution.
- `AsyncLocalStorage` requires disciplined context establishment at the HTTP and job boundaries.
- Platform operator workflows require a separate trusted design.
- Durable authorization audit records remain deferred.

## 7. Implementation Rules

- Never accept `userId`, `personId`, `organizationId`, role names, or permission codes from request payloads as authority.
- Always validate the session before membership selection.
- Always derive organization identity from the selected membership.
- Always verify that the membership belongs to the authenticated person.
- Always require active membership and active organization status.
- Always verify permission-evaluation membership and organization identifiers.
- Never treat an account context as organization context.
- Never let an authenticated user session create platform context.
- Never store request context in process-global mutable variables.
- Never continue when trusted request context is missing.
- Always authorize business operations through permission codes rather than role names.

## 8. Deferred Decisions

- Public request middleware, guards, and decorators.
- Membership-selection cookies or headers.
- Context caching and invalidation.
- Department, branch, assigned-record, self, and other permission scopes.
- Trusted platform operator context.
- Service-account and machine context.
- Background-job context.
- Durable authorization-denial audit persistence.
- Distributed tracing integration.
